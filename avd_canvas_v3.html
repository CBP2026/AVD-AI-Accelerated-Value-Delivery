BEYOND AGILE
Accelerated Value Delivery — A Framework for the Age of AI
CP  •  Draft v3.0  •  June 2026

Abstract

Agile methods were designed for a world where building software was slow, expensive, and divided across specialist silos. That world no longer exists. AI-powered development tools now allow a single product manager — with no prior development background — to design, build, and deploy working applications in a fraction of the time a traditional team would take.

This paper proposes Accelerated Value Delivery (AVD): a framework that compresses the build cycle aggressively while preserving the governance, quality controls, and accountability that enterprise software demands. AVD is stack-agnostic by design. It assumes the AI builds in whatever stack the approved toolchain supports, with the understanding that — in OutSystems-led organisations — output is converted into the platform via tooling under development, or generated directly in the platform when that capability lands.

1. The Problem with the Current Model

Traditional software delivery — exemplified by platforms like OutSystems — follows a long, sequential lifecycle. Discovery, design, sprint planning, development, QA, release, and post-production support each consume weeks. Specialists work in lanes: business analysts, architects, developers, designers, testers, product owners. Every handoff between lanes is a tax on time and clarity.

The metric that suffers most is Time to First Value. The customer waits months before seeing working software. By the time delivery arrives, the original problem has often shifted.

The question this paper asks is whether any of this still needs to take as long as it does.

2. The AI Inflection

AI development tools have changed what a single person can accomplish. A PM with product knowledge and AI fluency can deliver a functional MVP in a fraction of the time previously required. This breaks the core assumption that underpinned agile — that building software requires a team of specialists working in coordinated sprints over extended periods.

AI does not only accelerate coding. It accelerates architecture review, UX generation, documentation, testing, security scanning, and compliance reporting. The bottleneck is no longer technical capacity. It is decision-making, governance, and process design.

AVD is the framework that takes this shift seriously.

3. Stack-Agnostic by Design

The build mechanics described in this paper are deliberately stack-agnostic. The AI generates working software in whatever stack the approved toolchain supports. The governance, gating, and continuous-evolution mechanisms apply equally regardless of the underlying technology.

In OutSystems-led organisations, the assumption is one of two paths: either the AI-generated build is converted into OutSystems modules through tooling currently being developed by the R&D team; or, once the platform supports it natively, the build is generated directly in OutSystems. The framework's logic does not depend on which path applies.

4. Framework Overview

AVD has three phases connected by one continuous artefact — the Living Backlog. Phase 1 is PM-led inception and accelerated build. Phase 2 is a compressed technical gate. After Phase 2, the application is deployed and enters the organisation's standard support and operations cycle. The PM has no further role in run. Phase 3 is the continuous evolution model, powered by stakeholder sandboxes.

This phase boundary matters: AVD accelerates the build, not the run. Operations, SRE, incident response, and tiered support are unchanged. AVD does not attempt to redefine them.

5. Phase 1 — Inception & Accelerated Build

Phase 1 begins with discovery and ends with a working build handed to Phase 2. Discovery is not a precursor that happens elsewhere — it is the first activity of Phase 1, and the Vision Brief it produces is the signal that the build can start.

Context Check

Before discovery proper, a short context check determines the right build path. There are three paths:

PathScenarioArchitecture Treatment
Existing AppBuilding on or extending a live applicationArchitecture review is mandatory before build starts. PM flags decisions for Phase 2; does not make them.
Greenfield with ContextNo existing app, but the customer has real requirementsMinimal architecture sketch (hosting model, auth approach, data model shape) before build. Kept deliberately small.
Vanilla GreenfieldNo app, no constraintsBuild immediately. Architecture is formalised in Phase 2.

Discovery and the Vision Brief

The PM meets directly with the business owner — not a business analyst, not a proxy. The conversation is recorded. AI processes the recording and extracts candidate backlog items, which the PO reviews and approves. This seeds the Living Backlog.

The conversation produces a single artefact: the Vision Brief. One page. It captures the problem, the business goals, the success metrics, and the scope boundary. The PO signs it before the build starts. This is the contract and the start of the audit trail.

UX Grounding

UX in Phase 1 is about constraints, not screens. If a design system exists, the PM uses it as a reference from the start. If the project is true greenfield, a UX specialist defines grounding rules — components, navigation patterns, accessibility standards — in a single short session. The AI then builds within those rules. Detailed flow review is a Phase 2 activity.

The Build

The PM builds using the approved AI toolchain. No user stories are written; no wireframes are handed to designers. The PM uses AI-assisted QA tools continuously throughout the build. Scope is kept honest by the Vision Brief, which the AI references when flagging directions that drift from the agreed objectives.

Acceptance criteria are not lost. They are anchored in the Vision Brief at the outcome level and refined per feature inside each commit summary (see below). For future maintainers, AI regenerates technical documentation from the current code on demand, so the build remains comprehensible after the PM moves on.

Sign-off and Audit Trail

When the PM considers a feature complete, they trigger a commit summary. The system generates the summary automatically — what was built, the Living Backlog item it satisfies, the tests that passed. The PO reviews and approves, with a timestamped record. This mechanism is the audit trail. It is defined here and referenced — not redefined — elsewhere in this paper.

Quality

AI-assisted QA runs throughout the build. A human QA specialist performs a focused spot review before handoff to Phase 2. This combination catches defects earlier than a traditional post-sprint cycle without requiring a full-time QA presence during the build.

Minimum Team

RolePresenceResponsibility
PM (AI-First)Full-time during buildPrimary builder. Owns the build, triggers commit summaries, manages scope against the Vision Brief.
Product OwnerDaily reviewSigns Vision Brief. Reviews commit summaries. Final decision authority on scope and acceptance.
Technical CompanionOn-call (existing-app path only)Available to answer architecture questions. Does not build.
QA SpecialistFocused spot review before Phase 2 handoffIndependent human pass to complement AI-assisted QA run throughout the build.
AVD CoachFirst project onlyOwned by a central enablement function. Lightweight involvement on subsequent projects, not absorbed by the PO.

Output of Phase 1: a working application, a signed Vision Brief, a seeded Living Backlog, and a timestamped commit-summary audit trail.

6. Phase 2 — Technical Gate & Hardening

The MVP passes through a formal technical gate before going to production. This is non-negotiable for enterprise software. The gate is radically compressed using AI and runs as parallel tracks, all coordinated by the PM.

Parallel Tracks

TrackOwnerActivity
Security ScanningAutomated (triggered on handoff)Vulnerabilities, suspicious API calls, data-handling issues.
Architecture ReviewSenior Architect (by exception)Validates against the pattern library. By default, conforming builds pass automatically; only non-conforming or novel patterns require architect time.
Compliance DocumentationAI-generatedSecurity reports and compliance documentation produced automatically from the build.
UX Flow ReviewUX SpecialistEvaluates flows, interactions, usability. Parallel track — findings go to the Living Backlog unless they are deployment blockers.

Architect Capacity — Review by Exception

Architects are scarce and expensive. AVD does not assume an architect reviews every project in detail. The default is automated validation against the pattern library: builds that conform pass without architect time. Architects engage only on novel patterns, non-conforming structures, or genuinely new territory. This is what makes Phase 2 sustainable at portfolio scale.

Pattern Library and the First-Project Reality

The pattern library is owned by the Architecture Review Board and grows incrementally with each project. On the first AVD project, the library is effectively empty: there is no fast lane, and the architect uses standing architecture principles to review the build. The output of that project seeds the library. This first-project cost is real, and it is honest to name it. From the second project onwards, the fast lane begins to do work.

Tiered Blocker Model

TypeDefinitionAction
Deployment BlockerSecurity vulnerabilities, data exposure risks, critical compliance gapsBuild returns to the PM for resolution. Re-enters the gate when fixed. Not treated as a new project.
Non-BlockerUsability improvements, minor technical debt, non-critical findingsAdded to the Living Backlog with a remediation item. Does not delay deployment.

Only true blockers extend the timeline. Everything else ships with a flagged remediation item. This preserves momentum while keeping the safety bar non-negotiable.

Deployment Authority

AVD replaces the traditional change-advisory bottleneck with a model of pre-delegated deployment authority. The combined output of Phase 2 — passing security scans, architecture validation, and compliance documentation — is itself the deployment authority within a defined risk envelope. Out-of-envelope changes (new external integrations, regulated-data changes, new authentication surfaces) trigger an asynchronous standing-authority review rather than a scheduled meeting. Governance is preserved; the calendar wait is removed.

Output of Phase 2: production-ready application, deployable through the standard release path. Living Backlog updated with non-blocker findings.

7. Handoff to Run

AVD ends at deployment. The PM is the builder, not the operator. Once the application is in production, it enters the organisation's standard support and operations cycle: SRE, incident response, tiered support, change management for routine maintenance. AVD does not redefine any of this.

This boundary protects two things at once. It prevents the PM from becoming a long-tail support owner, which would invalidate the model. And it ensures that the application is supported by people whose discipline is operations, not build.

Continuous evolution — Phase 3 — is a separate cycle that runs in parallel with run, not a replacement for it.

8. Phase 3 — Continuous Evolution via Stakeholder Sandboxing

Once the product is live, the traditional PO bottleneck — all new requirements funnelled through one person writing user stories — is replaced by a decentralised, community-powered model. The Living Backlog and the sandbox model work together to drive continuous evolution while the application is supported under standard operations.

Sandbox Provisioning

  • A certified stakeholder submits a lightweight request describing what they want to build and which feature area it touches.
  • IT provisions a sandbox using a standard template, within a committed SLA.
  • Sandbox cost is pre-approved at programme level. No per-sandbox budget approval.
  • Every sandbox is automatically provisioned with synthetic data (see below).

Synthetic Data

No real production data may exist in any sandbox under any circumstances. Sandboxes run exclusively on synthetic data that mirrors the structure, volume, and behaviour of production without containing real personal information.

Building this synthetic data layer is a strategic, programme-level investment, not a per-project cost. For complex enterprise schemas, with foreign-key integrity, business rules, and edge cases, it is a multi-month engineering effort. Phase 3 cannot operate without it. Phases 1 and 2 can run in parallel with the layer being built, provided test data is handled with care.

Building and Submitting Features

Certified stakeholders build inside their isolated sandbox using AI tools. When ready, they submit the feature for community review. The PO defines the eligible voter group for each feature based on direct use or direct impact — not job title or seniority.

Voting and PO Decision

Votes are a prioritisation signal, not a mandate. The PO reviews highly voted features against business strategy, available resources, and the roadmap, and makes the final call. When a feature is rejected, the reasoning is published. Rejected ideas are archived in the Living Backlog with context — not deleted.

Approved features then move through the same Phase 2 gate that governed the original build. The same automated/exception model applies, so the gate scales with feature volume rather than collapsing under it.

Automated Agents

AgentFunction
Duplicate DetectorContinuously scans active sandboxes, flags similar proposals, and suggests merges to prevent duplicate effort.
Strategy MonitorWhen business objectives change, surfaces previously rejected ideas that are now relevant and flags them to the PO.
Auto-ArchiveArchives inactive sandboxes according to the organisation's inactivity policy, keeping infrastructure costs manageable.

Output: continuous, governed product evolution with full traceability and no lost ideas.

9. The Living Backlog

The Living Backlog is the connective tissue of AVD. It is not a sprint backlog. It is a continuously maintained, AI-assisted record of every idea, request, and finding across all three phases, with full attribution and context. Nothing is ever lost. Every item has a timestamp, an originator, and the context it came from.

SourceHow it enters
Phase 1 DiscoveryAI processes the recording and extracts candidates. PO reviews and approves.
Phase 2 Non-Blocker FindingsTechnical, UX, and security findings that do not block deployment are added automatically with remediation context.
PO-Initiated AdditionsThe PO can add items at any time, with context and attribution.
Phase 3 Rejected ProposalsFeatures voted down or rejected by the PO are archived with the reason.

10. Governance Model

AVD does not eliminate governance. It modernises it. Accountability is clearer, decision-making is faster, and the audit trail is built into the build process itself rather than maintained as a separate documentation layer.

RoleTraditional ModelAVD Model
Product OwnerBacklog manager, story writerDecision authority. Daily reviewer. Sign-off owner on commit summaries.
Business AnalystRequirements intermediaryEliminated. Replaced by direct PM-to-stakeholder conversation.
Architect / Tech LeadMulti-week reviewCompressed gate. Reviews by exception against the pattern library.
QA TesterPost-sprint testingAI-assisted QA continuous during build. Human spot review before Phase 2 handoff.
UX DesignerUpfront wireframingGrounding rules in Phase 1 (greenfield only). Flow review in Phase 2.
PM / Engagement ManagerCoordinator, facilitatorPrimary builder in Phase 1. Coordinator in Phase 2. Hands off at deployment.
AVD CoachN/AFirst project only. Owned by a central enablement function. Lightweight on subsequent projects.

Decision Hierarchy

  • The PO has final authority over what reaches production.
  • For stakeholder conflicts, the PM provides an AI-generated impact analysis; the PO decides.
  • Senior leadership can override, but the reasoning is published. Every decision is visible and accountable.

11. Failure Modes

AVD is presented honestly only if it names the failure paths. The framework defines a clear response for each.

Phase 2 finds a deployment blocker

The build returns to the PM. The same tools that produced it produce the fix. It re-enters the gate when ready. This is not a new project; it is a continuation of the same one.

PO rejects the MVP on completion

The Vision Brief was wrong, or scope drifted. The project re-enters discovery. Because no large team was disbanded, the cost is small. This is one of AVD's structural advantages: the cost of a failed build is contained.

A sandbox feature fails community vote

It is archived in the Living Backlog with the reason. The Strategy Monitor agent will resurface it if circumstances change.

Synthetic data missed a real-world bug

The bug enters the standard production incident path. The defect becomes a new case for the synthetic data layer, which improves over time. This is expected and budgeted for, not a framework failure.

The PM is unavailable mid-build

AVD requires a named backup PM with equivalent toolchain certification. The build state lives in the commit summaries, the Vision Brief, and the code itself — not in tribal knowledge.

13. Training and Certification

Not every stakeholder will participate in the sandbox model. Self-selection is healthy — only motivated people will invest in the learning. All sandbox participants complete mandatory training before being granted access. This mirrors existing corporate certification models such as Power BI or Snowflake.

Training covers building features responsibly with AI tools, security awareness specific to sandbox environments, how the commit-summary and voting processes work, and the boundaries of what can be changed within a sandbox. Experienced participants also acknowledge the new way of working in writing.

14. Security Model

The decentralised sandbox model only works with a layered security architecture. Six layers work together to ensure that openness does not compromise the production system.

LayerMechanism
1. IsolationSandboxes are fully isolated from production data and infrastructure.
2. Synthetic DataAll sandboxes use synthetic data only. GDPR and HIPAA compliant by design.
3. Automated ScanningAll sandbox code is scanned for security vulnerabilities before community review.
4. Role-Based PermissionsSensitive areas (authentication, payments, data exports) are restricted to privileged roles.
5. Architecture GateAny sandbox feature heading to production passes through the Phase 2 gate.
6. Audit TrailsEvery change in every sandbox is logged with full who/what/when/why traceability.

16. Empirical Basis and Pilot Path

The framework is informed by recent practical experience: in a short window, the author built and deployed several functional applications using AI tooling, with no prior development background. These applications demonstrated that the build mechanics described in Phase 1 are achievable in practice.

That experience is, deliberately and honestly, not enterprise validation. The applications were small in scope, lightly integrated, and not exposed to the full weight of regulated production. Their value is to demonstrate feasibility of the build model, not to substitute for an enterprise pilot.

The next step is a real pilot, scoped against an actual OutSystems engagement, with all of the prerequisites in section 12 in place. The pilot is where AVD's claims about quality, safety, and acceleration are tested under enterprise conditions.

17. A Note on Commercial Contracting

Customer engagements written in sprint-based Agile vocabulary do not map cleanly to AVD. Milestone billing tied to sprint outputs is the wrong instrument for a build that compresses radically. The shift this implies — from sprint-deliverable contracts to outcome-based contracting — is already being explored at OutSystems. It is mentioned here only to acknowledge the dependency; the contracting model is out of scope for this paper.

18. Closing

Agile served the industry well for three decades. Its values — speed, customer collaboration, working software over documentation, responding to change — are preserved and amplified by AVD. What is discarded is the bureaucratic scaffolding that slowed agile down once its founding assumptions were no longer true.

AVD is a framework in which a PM with AI tools is the primary builder, the product owner is a decision authority rather than a story writer, quality is continuous rather than sequential, and the community drives evolution after launch rather than a single gatekeeper. The Living Backlog ensures nothing is lost. The tiered blocker model ensures nothing unsafe ships. The commit-summary audit trail ensures everything is traceable. The handoff to run ensures the PM is not a long-tail operator.

It is not a framework built for the world agile was designed for. It is a framework built for the world that already exists.

— CP, June 2026

Annex A — Comparison with Existing Frameworks

BMAD preserves the structure of Agile (stories, sprints, documents) and automates the generation of each role's output with AI. AVD is more radical: it eliminates stories as an intermediate layer and treats the request itself as the source of truth.

SAFe scales Agile up; AVD compresses it down. The two are not directly comparable, but the contrast clarifies AVD's position.

Shape Up replaces sprints with six-week cycles and betting tables. AVD shares Shape Up's rejection of fixed sprint cadence, but is built around AI-led build rather than small autonomous teams.

TraceabilityStory → task → codePRD → agent output → codeRequest → commit summary → code
Sprint structureRequiredPreservedEliminated
Continuous evolution modelPO-mediatedPO-mediatedCommunity sandboxes + Phase 2 gate

Alternatives to Jira (DevRev, BridgeApp, Linear, Plane) are all still centred on tasks and tickets. None implements the "request as source of truth" model that AVD proposes. This is the white space in the market that AVD occupies.
